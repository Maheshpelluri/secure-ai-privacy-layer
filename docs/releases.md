# Releases

This repository publishes architectural and research milestones rather than
software releases.

## Versioning

- Versions follow semantic-style numbering for repository publication
  milestones.
- `v0.1.0` marks the initial public architecture publication.
- Future versions should reflect meaningful repository-level changes, such as a
  new phase, a major architecture revision, or a validated prototype milestone.

## Release Policy

- Release notes should be derived from `CHANGELOG.md`.
- Release tags should only be created when the repository state is coherent and
  reviewable.
- No release should imply a working product unless code and validation evidence
  exist.

## Architecture Releases

- Use for major architecture publication updates or clarified design direction.
- Keep scope limited to conceptual content, research framing, and repository
  governance.

## Prototype Releases

- Use only after a reference implementation exists.
- Include the relevant implementation summary, validation notes, and any known
  limitations.

## Production Releases

- Reserve for a mature, supported system with documented testing, security
  review, and operational expectations.
- Production releases should not be implied by documentation-only updates.
